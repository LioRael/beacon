# ADR 0004: Execute the Skills CLI through an npm package runner

Status: accepted during the Agent Skill package-management grilling session.

Beacon delegates third-party Agent Skill operations to the `skills` npm package through a package runner rather than a globally installed `skills` executable. Beacon prefers `npx --yes skills@^1.5.18` and falls back to `bunx skills@^1.5.18`; neither path installs a global command, though the selected runner may populate its normal package cache. If neither runner can resolve and execute the package, the manager is unavailable with explicit setup guidance. Individual Agent Skills are inventory items, while the resolved `skills` package is their external manager. Agent Skill detection, diagnosis, and updates participate in Beacon's existing main commands rather than a separate `skills` command group. Main checks aggregate global Skills and Skills belonging to the current project, omitting the project scope when the working directory has no project Skill state. Inventory IDs are `skill:global:<name>` and `skill:project:<name>`; a bare name is accepted only when it uniquely identifies one item in the current result.

Because `skills update` has no read-only or machine-readable planning mode, Beacon detects updates by copying the relevant Skill state into an isolated temporary home or project, running the package-resolved `skills` updater there, and comparing content hashes before and after. Detection never writes the user's Skill directories and never infers results from human terminal text; only a confirmed Beacon upgrade runs the updater against real state. The compatible package range is part of the reported action, so confirmation does not switch to an unbounded future major version.

Detection invokes the isolated updater once per Skill with an explicit scope and target, using the existing maximum of four independent checks concurrently. Exit status and before/after state therefore belong to one inventory item; failures remain attributable without parsing human output, even when checking Skills from the same source separately costs additional remote requests.

A Skill revision is the SHA-256 content hash of its directory, exposed in full in machine output and abbreviated only in human output. Skill updates use Beacon's rolling verification mode: the installed revision must change after an update, while a source that advances after confirmation may produce a newer revision than the one observed during detection.

Outdated Skill reports include a file-level change manifest derived from the isolated before and after trees. Machine output returns every relative path as added, modified, or removed, and interactive confirmation shows the same paths and counts. Beacon does not emit file contents or a textual diff.

The `skills` inventory follows Beacon's existing inventory selection model. Fresh configuration and catalog migration enable it only when `npx` or `bunx` can resolve the compatible package and pass the capability probe; an unselected manager stays silent. If a user explicitly keeps the inventory enabled and both runners later become unavailable or cannot resolve the package, Skill inspection fails in isolation and the overall command returns a partial result with package-runner guidance.

The initial compatibility window is `skills >=1.5.18, <2.0.0`, expressed to package runners as `skills@^1.5.18` and verified again from runtime output. Beacon probes the selected runner version, the resolved Skills CLI version, and its machine-readable list capability. It records the runner path, runner version, package constraint, and resolved Skills version for preflight and post-update drift checks. An incompatible result or failed capability probe falls through from `npx` to `bunx`, then produces an isolated inventory failure rather than speculative parsing or execution.

Beacon treats the Skills CLI's global `.skill-lock.json` and project `skills-lock.json` as read-only manager receipts. The initial adapter recognizes global receipt version 3 and project receipt version 1, never writes either file directly, and fails closed with compatibility guidance when it encounters an unknown format. All authoritative receipt writes remain Skills CLI operations.

Every package-runner and Skills CLI subprocess launched by Beacon receives `DISABLE_TELEMETRY=1`. A Beacon read or update must not silently emit third-party usage events; this does not alter telemetry behavior when the user invokes either tool directly.

Private Skill sources are supported only through authentication already present in the invocation environment, such as GitHub token variables, an SSH agent, or Git credential environment. Beacon never discovers, prompts for, copies, or persists credentials. Authentication material is excluded from machine output, logs, history, and source locators; an authentication failure remains isolated to the affected Skill inventory result.

Agent Skills extend the schema v2 inventory report additively with scope, installation source, a credential- and home-redacted source locator, and update manager. Existing current, latest, and action fields remain unchanged. Homebrew inventory items populate the same metadata, so main-command summaries and confirmations no longer hard-code an inventory's source or updater.

Beacon resolves the current Skill Project by walking upward from the working directory to the nearest `skills-lock.json`. If no lock exists, it uses the current Git worktree root and asks the Skills CLI for project items there. If neither boundary exists, the operation has no project Skill Scope and inspects only global Skills; discovery never walks above the resolved project boundary.

Installed Skills that the Skills CLI can list but cannot safely update remain visible as unmanaged inventory items. Local sources, `node_modules`-synchronized items, legacy installs without tracking metadata, and other entries without a reliable source produce no update action; ordinary checks remain successful, while `doctor` explains the specific limitation and manual recovery path.

Project Skill updates are target-scoped and preserve local work. A current content hash that differs from the lock revision, or an uncommitted Git change overlapping the canonical target Skill or `skills-lock.json`, makes that item unmanaged and removes its update action. Unrelated worktree changes do not block it, and the initial feature has no force-overwrite path. Beacon does not inspect agent-specific copies or links for this guard.

Global Skills use the local observation baseline described below because the global receipt stores a remote Git tree identity rather than an installed-content hash. Git worktree checks apply only to project-scoped items.

For global receipts whose manager revision is not a local content hash, Beacon uses a local observation baseline keyed by the receipt fingerprint. The first observation trusts the current content and cannot discover older edits; later content changes under the same receipt are locally modified, while a receipt change establishes a new baseline. The baseline is supplementary local evidence and never replaces or rewrites the Skills CLI receipt.

The Skills CLI exclusively owns agent-specific copies, links, and installation topology. Before a confirmed update, Beacon keeps a recovery copy of only the canonical Skill directory and applicable lock file; it does not enumerate or automatically restore agent-specific paths. A command or post-update verification failure stops later targets, preserves the recovery copy in Beacon state, and reports its home-redacted path for manual recovery. Successful verification discards the copy.
